Ftarg — Ch.3: The Management Layer — Bot, Database, Automation

  • python
  • telegram-bot
  • automation
  • devops

Protocol is solid (Ch.1). Stack is running (Ch.2). Now the part that sits on top — user management and automation.

Built a Telegram bot from scratch to handle everything user-facing. Handles the full lifecycle: request, approval, link delivery, cleanup.

Why a custom bot

Marzban has its own Telegram bot, but it's admin-only. No user-facing flow.

I needed: user sends a request → admin approves or rejects → user gets their link. Plus encrypted storage, auto-cleanup, preset rejection reasons. None of that exists in Marzban's bot.

So now there are two bots. Marzban's for panel management. Custom one for users.

User flow

User: /start
  └── bot explains the service
 
User: /request
  └── bot creates entry, stored in encrypted DB
  └── admin gets notified
 
Admin: /pending
  └── sees list of pending requests
  └── approve or reject
 
Approved:
  └── bot creates user in Marzban via API
  └── user gets notified
  └── /mylink returns VLESS subscription link
 
Rejected:
  └── admin picks preset reason
  └── user gets notified
  └── request cleaned from DB

User side: /status, /mylink, /guide. Admin side: /stats, /health, /check, /relink, /delete, /sync, /purge, /logs.

What the bot does

  • Encrypted storage. Request data lives in an encrypted file. Fernet (AES-256-GCM).
  • Marzban API integration. Bot talks to Marzban via REST — creates users, fetches links, syncs state.
  • Retry logic. Every API call wrapped with retries and exponential backoff. Docker hiccups happen, users never notice.
  • Auto-cleanup. Stale requests cleared on a schedule. Bot DB stays in sync with Marzban panel.
  • Auto-delete messages. Old bot replies get removed on the next interaction. Chat stays clean.
  • Monitoring + backup. Health check and DB backup run on cron. Single-node scope — nothing fancy.

The whole stack

+------------------------------------------------------------------+
|                                                                  |
|   Ch.1 — PROTOCOL                                                |
|   VLESS + REALITY + TLS 1.3 impersonation                        |
|                                                                  |
|   Ch.2 — INFRASTRUCTURE                                          |
|   Marzban + XRay in Docker (host network)                        |
|   WARP outbound via XRay native WireGuard                        |
|                                                                  |
|   Ch.3 — MANAGEMENT LAYER                                        |
|   Telegram bot, encrypted DB, Marzban API                        |
|   Auto-cleanup, monitoring, backups                              |
|                                                                  |
+------------------------------------------------------------------+

Protocol handles transport. Infrastructure handles the runtime. Management layer handles users. That's the whole build.
