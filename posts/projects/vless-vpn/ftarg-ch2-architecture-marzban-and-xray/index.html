Ftarg — Ch.2: Architecture — Marzban, XRay, and the Routing Problem

Previous chapter covered why most protocols get caught by DPI and how REALITY avoids that. This one is about the actual stack — what's running and why.

Marzban

Marzban is an open-source management panel for XRay. Handles user creation, subscription links, traffic tracking, expiry. Has a built-in Telegram admin bot.

Why Marzban over 3x-ui or others:

  • Cleaner API
  • Docker-native
  • Better multi-node support in its architecture
  • You configure XRay directly

It's not the proxy. It's the control panel. XRay does the actual work.

Docker setup

Marzban runs in Docker with host network mode. Improtant part is the REALITY needs direct access to port 443 for the TLS handshake. Docker's default bridge networking adds a NAT layer in between — the handshake gets mangled, connections behave unpredictably or just drop.

Host network mode = container uses the host's network stack directly. No NAT, no bridge. Port 443 works as expected.

Single compose file, XRay config mounted as a volume.

The routing problem

This was the biggest headache of the whole build.

What I needed

The proxy server accepts connections on port 443 (VLESS+REALITY). Traffic comes in, gets decrypted, needs to go out to the internet. Simple.

But I didn't want outbound traffic going from my server's real IP. Reasons: privacy, IP reputation, separation between the proxy identity and the underlying host. So I needed an outbound tunnel.

What I tried first

ProtonVPN. WireGuard-based, seemed solid. Set it up as a system-level VPN on the host. wg-quick up, routing table updated, traffic going through the tunnel. Looked good.

Then port 443 stopped accepting connections.

I spent a while trying to make it work — policy routing, ip rules, different configs. Couldn't get inbound and outbound to coexist. At that point I moved on and started looking at other approaches.

Why system-level VPN breaks inbound

Later I understood the actual problem. When you bring up a system-level VPN, it takes over the default route. Every packet leaving the host goes through the tunnel — including responses to incoming connections.

So, what happens:

  1. Client sends SYN to your real IP on port 443
  2. Your server accepts, sends SYN-ACK
  3. But SYN-ACK goes out through the VPN tunnel
  4. Client gets a response from a completely different IP
  5. Connection dropped

Classic asymmetric routing. The incoming path and outgoing path don't match.

BROKEN (system-level VPN):
 
Client --[SYN to real IP:443]--> Server
                                   |
              response routed ---> VPN tunnel ---> different IP
                                                      |
Client <-- [SYN-ACK from wrong IP] <-----------------+
         connection rejected

This is a fundamental conflict. A server that needs to accept inbound connections on a specific port AND route outbound through a tunnel — system-level VPN can't do both cleanly.

The fix: XRay native WireGuard outbound

Turns out XRay has built-in support for WireGuard as an outbound protocol. Instead of tunneling all system traffic, only the proxied traffic goes through it.

I switched to Cloudflare WARP — free WireGuard-based tunnel. But instead of running it as a system VPN, configured it as an outbound inside XRay.

System networking stays completely untouched. Port 443 inbound works normally. XRay handles the tunnel internally for proxy traffic only.

WORKING (XRay native WireGuard outbound):
 
Client --[443]--> Server (direct, normal routing)
                    |
                    +--> XRay
                           |
                           +--> proxy traffic --> WARP tunnel --> internet
                           |
                    +--> response back to client (correct source IP)

Clean separation. Inbound = direct. Outbound = tunneled. No routing conflicts.

The solution isn't obvious unless you know XRay can do WireGuard natively. Application-level routing instead of system-level routing — that's the key.

Current stack

+------------------------------------------+
|              Host (Fedora)               |
|                                          |
|  +------------------------------------+  |
|  |     Docker (host network mode)     |  |
|  |                                    |  |
|  |   Marzban panel                    |  |
|  |     └── XRay core                  |  |
|  |           ├── Inbound:             |  |
|  |           │   VLESS+REALITY (443)  |  |
|  |           └── Outbound:            |  |
|  |               WireGuard (WARP)     |  |
|  +------------------------------------+  |
|                                          |
|  DuckDNS (dynamic DNS)                   |
|  firewalld (443 only)                    |
+------------------------------------------+

Running on residential fiber with dynamic IP. DuckDNS handles DNS updates. Not ideal for production but fine for dev — residential IPs are less likely to be on blocklists, which is actually an advantage during testing.

Protocol layer handled. Stack is running. Next problem: making sure the operational side doesn't leak what the protocol layer protects.
