Proxy service. Ch.1: Why Most Proxy Protocols Fail Against Modern DPI

  • proxy
  • dpi
  • tls
  • network-security

Most VPN and proxy protocols weren't designed to be invisible. They were designed to be fast, secure, or easy to deploy. But when you put them in front of a modern DPI system, everything falls apart. Why is that?

How DPI identifies proxy traffic

DPI doesn't need to decrypt anything. It works on metadata — packet sizes, timing, handshake structure, header bytes, entropy patterns.

Three main techniques:

1. Protocol fingerprinting. Every protocol has a handshake signature. OpenVPN starts with a specific opcode byte (0x38 for P_CONTROL_HARD_RESET_CLIENT_V2). WireGuard opens with a fixed 4-byte message type. These are static, predictable, and trivial to match with a regex on the first packet. And DPI sees this VPN service.

2. Entropy analysis. Encrypted traffic has near-uniform byte distribution. Normal HTTPS also has high entropy, but it's wrapped in TLS with recognizable structure — certificates, extensions, ALPN negotiation. A raw encrypted stream without that framing (like Shadowsocks or early VMess) stands out. It's not HTTP, not TLS, not any known protocol — just high-entropy noise. That alone is a signal.

3. Statistical fingerprinting. Packet size distributions and timing patterns vary between protocols. A WireGuard keepalive is always 32 bytes. Shadowsocks handshake packets cluster around specific sizes depending on cipher choice. DPI systems build statistical models and flag traffic that matches known distributions, even if the content is opaque.

OpenVPN and WireGuard are the easiest to block. Their handshakes are essentially self-announcing — a single signature rule catches them. Some DPI systems don't even bother with deep analysis; they just match the first few bytes.

Shadowsocks improved on this significantly. No fixed header, no recognizable handshake. But it still produces a stream of high-entropy data with no protocol framing. Modern DPI detects this by exclusion — if it's not TLS, not HTTP, not SSH, not DNS, and it's high-entropy on a common port, flag it. The "looks like nothing" approach worked for years, but statistical detection caught up.

VMess has similar issues. The protocol adds its own framing, and the packet length distribution is distinguishable from real HTTPS under analysis. Later versions improved, but the fundamental problem remains: it's a custom protocol pretending to be nothing, in an environment where "nothing" is suspicious.

The TLS impersonation approach

The insight that changed everything: don't invent a new protocol. Instead, make proxy traffic look exactly like regular HTTPS to a real, legitimate website.

This is what VLESS + REALITY does.

A normal TLS 1.3 handshake:

Client                          Server
  |                               |
  |--- ClientHello (SNI: example.com) -->|
  |                               |
  |<-- ServerHello + Certificate ---|
  |                               |
  |--- Finished ----------------->|
  |                               |
  |<===== Encrypted tunnel ======>|

The SNI (Server Name Indication) field tells the server which domain the client wants. DPI systems read this to categorize traffic. If the SNI says example.com and the TLS handshake looks legitimate, DPI has no reason to interfere — it looks like someone browsing a normal website.

REALITY takes this further. Instead of using a self-signed certificate (which DPI can detect by checking against known CAs), it proxies the actual TLS handshake to the impersonated server. The certificate that comes back is real. The handshake is real. To any observer — including DPI — this looks identical to a genuine TLS 1.3 connection to that domain.

The proxy server only takes over after the handshake completes. The client authenticates using a pre-shared key embedded in the handshake, and the connection switches to the proxy tunnel. If someone without the key connects — including DPI systems doing active probing — they get served the real website. There's nothing to detect.

DPI perspective:
 
Client --> [TLS 1.3 ClientHello, SNI: legitimate-site.com] --> Server
       <-- [Real certificate from legitimate-site.com]     <--
       --> [Encrypted application data]                     -->
 
Verdict: Normal HTTPS. Pass.
 
Actual flow:
 
Client --> proxy server --> authenticates via pre-shared key
       --> tunnel established
       --> traffic routed through proxy

Why SNI target selection matters

Not every domain works as an impersonation target. The SNI target needs to:

  • Support TLS 1.3 (mandatory — REALITY only works with 1.3)
  • Support HTTP/2 — needed for the handshake to look complete
  • Be plausible for the client's geography and usage patterns
  • Be a high-traffic domain (so connections to it don't stand out)
  • Not use certificate pinning or other mechanisms that break the proxy handshake

Some popular targets fail silently. One well-known example — a major tech company's domain supports TLS 1.3 but its H2 implementation breaks the handshake flow. No error on the server side, but the connection drops. Took a while to figure that out.

The ideal target is a large, boring, high-traffic site that's accessed constantly from everywhere. Search engines, CDN providers, major web platforms. The kind of domain where one more TLS connection is invisible in the noise.

What this means for architecture

If REALITY solves the detection problem at the protocol level, the remaining engineering challenges are everything around it — routing, scaling, user management, operational security, and making sure nothing else in the stack leaks metadata that undoes the transport-level privacy.
